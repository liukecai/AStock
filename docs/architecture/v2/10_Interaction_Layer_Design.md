# 10 Interaction Layer Design

> AStock V2 交互层架构设计。
>
> 交互层（Interaction Layer）是介于 Web 视觉展示与后端大模型引擎之间的核心层。它的目标是将 AStock 从“单向数据展示工具”升级为“双向人机协同系统”。
>
> 前置阅读：[07_Web_Design.md](./07_Web_Design.md)（前端视觉与状态设计）、[04_LLM_Integration.md](./04_LLM_Integration.md)（大模型抽取机制）。

---

## 1. 核心设计目标

V2 系统的复杂性在于“图谱路径”和“事件逻辑”是动态且海量的。传统的点击查询无法满足所有需求，纯自动的 LLM 抽取又容易产生幻觉。

交互层设计目标：
1. **对话式交互 (Conversational UI)**：用自然语言降低复杂图谱查询的门槛。
2. **专家干预闭环 (Human-in-the-loop)**：允许人类专家审核、纠错、标注，反哺知识图谱。
3. **主动知识喂入 (Proactive Input)**：允许用户上传私有文档或新闻，即时触发系统推理引擎。
4. **动态监控预警 (Alerting)**：从被动查询转为基于事件流的主动通知。

---

## 2. 对话式查询 (Conversational Querying)

传统的筛选框（如：选择行业、事件类型、日期）将被“自然语言 ↔ 图谱查询”引擎（NL2Graph）增强。

### 2.1 典型场景
- **意图解析**：用户输入：“最近有哪些涉及半导体材料的供应短缺事件？”
  - 交互层调用内部小模型，将意图解析为结构化查询：`event_type = supply_shortage`, `entity_type = Industry`, `entity_name = 半导体材料`。
- **归因追问**：在展示了股票推荐后，用户追问：“中船特气这条路径上的证据是什么？”
  - 交互层携带当前上下文（Context），调用 `Explain API` 生成具体的证据引用文本。

### 2.2 架构约束
- **严禁直接让 LLM 查数据库**：所有自然语言必须先转化为标准的 API 调用或 GraphQL/Cypher 查询语句。
- **可解释性底线**：对话系统回复的内容，必须带有图谱关系节点和文档来源的超链接。不允许 LLM 凭空捏造回答。

---

## 3. 专家干预与纠偏闭环 (Human-in-the-loop)

知识图谱的质量决定了推理的上限。交互层必须提供高效的质量控制流。

### 3.1 图谱纠错反馈 (Feedback Loop)
在 `Company Knowledge Card` 或 `Supply Chain Explorer` 页面中，用户可以对任何实体或边进行纠错干预。
- **操作类型**：`降权 (Downvote)`、`报错 (Report Error)`、`补充证据 (Add Evidence)`。
- **底层影响**：系统调用图谱更新 API，记录审计日志 `operator = manual_admin`，并动态更新 `confidence` 或触发重审流程。

### 3.2 高效审核工作流 (Review Workflow)
对于 LLM 抽取在 `candidate_relations` 中的低置信度数据，交互层提供“Tinder式”的审核卡片：
- **展示**：左侧展示原始文本（Evidence），右侧展示 LLM 抽取的结构化实体/关系。
- **操作**：专家一键点击 `Approve`（合入正式图谱）、`Reject`（废弃）或 `Edit`（修改后合入）。

---

## 4. 动态订阅与预警 (Alerting & Subscription)

量化研究需要“事件驱动”，交互层负责将静态的图谱发现转化为动态的行情信号。

### 4.1 订阅维度
- **实体订阅**：关注特定节点（如“六氟化钨”或“台积电”），当有新事件影响到该节点时触发通知。
- **逻辑订阅**：关注某种传播路径（如“任何导致算力板块成本上升的政策事件”）。
- **标的订阅**：关注具体的自选股池。

### 4.2 推送阈值设置
用户可在交互层设置触发条件，避免信息轰炸：
- 仅当推理结果的 `final_score > 80` 且 `confidence > 0.85` 时推送。
- 仅当某事件导致某股票发生“首次多跳关联”时推送。

### 4.3 推送渠道
交互层可以解耦输出到 Webhook、钉钉/飞书机器人、Telegram Bot 或是邮件，不仅仅局限于 Web 站内信。

---

## 5. 主动知识喂入 (Proactive Knowledge Input)

打破系统被动定时爬取新闻的局限。

### 5.1 工作流
1. **拖拽解析**：用户在前端直接上传一份最新的 PDF 研报，或者粘贴一段刚看到的非公开路演纪要。
2. **即时抽取**：交互层触发 [04_LLM_Integration.md](./04_LLM_Integration.md) 定义的抽取 Pipeline。
3. **即时推理**：生成候选关系并立即执行一次 On-demand Reasoning（详见 03 推理引擎运行时）。
4. **即刻反馈**：数秒内向用户展示这份私有纪要会影响图谱中的哪些节点，以及可能利好/利空哪些股票。

---

## 6. 架构决策记录（ADR）

### ADR-001：对话交互不直连数据库
所有自然语言查询（NL2Graph）的输出必须是标准的系统 API 参数，再由后端服务执行查询，以确保数据权限安全和查询性能可控。

### ADR-002：区分“机器候选”与“专家事实”
通过交互层进入的纠错和审核操作，其 `confidence` 将被锁定为最高级别（如 0.99 事实级），后续的自动 LLM 抽取不得覆盖人工审核通过的关系。

### ADR-003：私有知识空间隔离
当用户上传私有文档（主动喂入）时，抽取的图谱关系如果未被公共数据源证实，可打上 `user_private` 标签。在多租户场景下，防止用户私密知识泄露给全局图谱。
