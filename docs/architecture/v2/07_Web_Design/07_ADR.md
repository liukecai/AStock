# 07 ADR

> 本文档记录 AStock V2 Web Design 阶段的关键架构决策。

---

## ADR-001：Web 层以解释为核心，而不是只展示排名

### 状态

Accepted

### 背景

V1 更偏向展示结果列表。V2 需要让用户理解事件、图谱路径、证据和验证之间的关系。

### 决策

Web 层优先展示：

```text
事件
路径
证据
分数拆解
验证结果
```

再展示排名。

### 影响

优点：

- 可解释性更强。
- 用户更容易复核。
- 更符合 V2 事件推理定位。

代价：

- 页面信息密度增加。
- 前端组件更复杂。

---

## ADR-002：路径图作为核心交互组件

### 状态

Accepted

### 背景

事件与标的之间的关系往往不是直接关系，而是多跳路径。

### 决策

Event Impact Path 和 Reason Path Graph 作为核心组件。

### 影响

优点：

- 用户能直观看到推理链路。
- 图谱价值更明显。

代价：

- 需要控制节点数量和布局复杂度。

---

## ADR-003：低置信结果必须显式展示

### 状态

Accepted

### 背景

部分关系来自弱证据或候选关系，如果不提示会误导用户。

### 决策

所有低 confidence、pending_review、weak_evidence 状态必须在 UI 中明确标记。

### 影响

优点：

- 降低误读风险。
- 提高系统可信度。

代价：

- UI 状态更多。

---

## ADR-004：验证结果必须展示样本数

### 状态

Accepted

### 背景

没有样本数的胜率和平均收益容易被误读。

### 决策

所有验证统计都必须展示 sample_count。

### 影响

优点：

- 用户能判断统计可靠性。
- 避免小样本误导。

代价：

- 页面需要更多解释信息。

---

## ADR-005：Web 不承载业务推理逻辑

### 状态

Accepted

### 背景

如果前端执行推理，会导致逻辑分散和结果不一致。

### 决策

Web 只消费 API DTO，不直接访问数据库，不执行图谱推理，不计算最终分数。

### 影响

优点：

- 前后端职责清晰。
- 结果一致。
- 易于测试。

代价：

- 后端 API DTO 需要更完整。

---

## ADR-006：空状态和错误状态必须说明原因

### 状态

Accepted

### 背景

V2 数据链路较长，用户需要知道为什么没有结果。

### 决策

空状态、错误状态、数据过期、低置信、缺失数据必须有明确提示。

### 影响

优点：

- 用户体验更清晰。
- 便于排查数据问题。

代价：

- 需要统一 UI 状态规范。

---

## 结论

Web Design 阶段的核心决策是：

```text
解释优先
路径可视化
证据可见
低置信提示
验证显示样本数
前端不做推理
```

这些决策保证 V2 Web 能准确呈现事件推理系统的价值。
