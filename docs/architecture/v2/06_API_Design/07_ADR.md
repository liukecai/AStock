# 07 ADR

> 本文档记录 AStock V2 API Design 阶段的关键架构决策。

---

## ADR-001：API 统一使用 /api/v2 前缀

### 状态

Accepted

### 背景

V2 API 与现有 V1 接口在语义和响应结构上会有明显差异。

### 决策

所有 V2 接口统一使用：

```text
/api/v2
```

### 影响

优点：

- 与旧接口隔离。
- 便于后续版本演进。
- 前端迁移更清晰。

代价：

- 需要维护新旧接口兼容期。

---

## ADR-002：API 返回 DTO，不直接返回 ORM 对象

### 状态

Accepted

### 背景

ORM 对象反映数据库结构，不适合作为前端契约。

### 决策

API 必须返回 DTO。

### 影响

优点：

- 隐藏数据库细节。
- 响应结构稳定。
- 便于前端使用。

代价：

- 需要维护 DTO 转换层。

---

## ADR-003：统一响应结构

### 状态

Accepted

### 背景

不同接口响应结构不统一会增加前端处理成本。

### 决策

统一使用：

```text
success
data
error
meta
```

### 影响

优点：

- 前端处理简单。
- 错误处理统一。
- 便于接口测试。

代价：

- 少量简单接口也需要包装结构。

---

## ADR-004：重型任务不在 API 请求中执行

### 状态

Accepted

### 背景

事件抽取、图谱更新和验证计算可能耗时较长。

### 决策

API 请求只读取已计算结果。

重型任务由后台调度或 Worker 执行。

### 影响

优点：

- API 响应稳定。
- 任务可重试。
- 成本可控。

代价：

- 数据存在一定延迟。

---

## ADR-005：Explain API 必须返回路径和证据

### 状态

Accepted

### 背景

如果解释接口只返回自然语言，用户无法复核推理过程。

### 决策

Explain API 必须返回：

```text
reason_paths
evidence_list
score_breakdown
validation_summary
```

### 影响

优点：

- 可解释性强。
- 支持前端路径图。
- 支持人工复核。

代价：

- DTO 更复杂。

---

## ADR-006：列表接口必须分页

### 状态

Accepted

### 背景

事件、证据、候选关系和验证结果会持续增长。

### 决策

所有列表接口必须支持 page 和 page_size。

### 影响

优点：

- 避免大响应。
- 提高前端性能。

代价：

- 前端需要处理分页状态。

---

## ADR-007：Graph Path 查询必须限制深度

### 状态

Accepted

### 背景

图谱路径查询如果不限制深度，会产生大量噪音和慢查询。

### 决策

Path API 默认 max_depth 不超过 4。

### 影响

优点：

- 查询可控。
- 结果更可解释。

代价：

- 可能遗漏远距离关系。

---

## 结论

API Design 阶段的核心决策是：

```text
版本化接口
DTO 输出
统一响应结构
重任务离线
解释接口返回路径和证据
列表分页
路径查询限制深度
```

这些决策保证 V2 API 稳定、可维护、可解释。
