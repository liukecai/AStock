# 09 Future V3

> AStock 从 V2 向 V3 的演进方向。
>
> V3 不是 V2 的必要组成部分。V2 完成后，根据系统规模和业务需求再决定是否以及何时演进。

---

## 1. 数据库演进

### 1.1 PostgreSQL + pgvector

V2 第一阶段使用 SQLite / PostgreSQL 关系型存储。当图谱规模增加或需要向量检索能力时，可引入 pgvector：

- 证据文本和公告文本的向量化存储。
- 基于语义相似度的证据检索。
- 事件和实体的语义匹配。

### 1.2 Neo4j

当图谱路径查询规模明显超过关系型数据库能力时，可引入 Neo4j：

- 复杂多跳路径查询。
- 图算法（社区发现、中心度计算、最短路径）。
- 知识图谱可视化工具链。

迁移策略：从 `kg_entities` / `kg_relations` 定期同步到 Neo4j，关系型数据库仍作为主存储。

---

## 2. GraphRAG

将 Knowledge Graph 与 RAG 结合，增强事件分析能力：

- 基于图谱结构的上下文构建（Graph-aware context）。
- 事件发生时自动检索相关产业链知识。
- 辅助生成更准确的个股解释。

---

## 3. Agent 自动维护图谱

引入自主 Agent 持续维护和扩充图谱：

- 定期扫描年报、公告、官网，自动发现新产品和新关系。
- 自动检测过期关系（业务剥离、产品停产）。
- 自动合并重复实体。
- 自动触发低覆盖度领域的定向补全。

---

## 4. 事件权重自学习

基于 Validation Loop 积累的历史数据，自动优化：

- 事件类型权重。
- 关系传播权重。
- 股票响应模型。
- 最优验证窗口选择。
- 噪音事件自动降权。

从固定规则权重演进为数据驱动的自适应权重。

---

## 5. 多市场扩展

从 A 股扩展到更多市场：

- 港股：同一 Company 多 Stock 建模。
- 美股：跨市场产业链关联。
- 商品期货：直接跟踪商品价格验证。
- 债券/基金：资产类型扩展。

图谱的 `Company → listed_as → Stock` 分离建模已为此预留空间。

---

## 6. 其他方向

- 多语言支持：英文新闻和公告的抽取与归一化。
- 实时事件流：从盘后批处理演进为准实时事件处理。
- 用户自定义图谱：允许用户添加私有关系和自定义事件类型。
- 组合分析：从单只股票扩展到事件影响组合的整体分析。

---

## 7. 演进原则

```text
V2 先跑通闭环
V3 再追求规模和智能
```

V3 的前提是 V2 的图谱、推理、验证闭环已经稳定运行并积累了足够的历史数据。
